# Abort-reason catalog — developer guide

Every time the simulation *declines* to launch an engagement — because ROE forbids it, the
target is out of envelope, the magazine is empty, a datalink is stale, and so on — it records
a **stable, machine-readable code** instead of a free-text string. Those codes are the
`Engagement|…|<CODE>` and `POLICY_DENIAL` rows in the order log, the `ENGAGE_ABORT` /
`POLICY_DENIAL` lines in the C2 message log, and the reasons the after-action review surfaces
to the player. Because they are pinned by replay goldens and read by the UI, the code set is a
**contract**, not an implementation detail.

This page explains where those codes come from, the manifest → codegen workflow that keeps
them in one place, the two-layer enum split, and how to add or change a code without breaking
replay determinism. The normative *catalogue* of terms lives in
[req 12 — Terms Glossary](../../Game-Requirements/requirements/12-Terms-Glossary.md); this is
the *engineering* reference for the machinery behind it.

---

## Single source of truth: the manifest

All abort codes are declared once in
[`data/glossary/abort_reason_manifest.json`](../../data/glossary/abort_reason_manifest.json)
(req 12). The manifest groups codes into **families**:

| Family | Kind | Backing enum | Where it fires |
|--------|------|--------------|----------------|
| `Doctrine` | enum-mapped | [`FireAbortReason`](../../src/ProjectAegis.Sim/Policy/FireAbortReason.cs) | Policy/ROE layer denials (`IPolicyEvaluator`, ADR-002) — `ROE_HOLD_FIRE`, `ROE_WEAPONS_TIGHT`, `WRA_RANGE`, `WRA_SALVO`, aspect blocks, `COMMS_DENIED` |
| `Engage` | enum-mapped | [`EngagementAbortReason`](../../src/ProjectAegis.Sim/Engage/EngagementAbortReason.cs) | Engagement pipeline (`MvpEngagementResolver`) — `OUT_OF_ENVELOPE`, `DLZ_OUT`, `NO_AMMO`, `AIR_NOT_READY`, `CYBER_SPOOF_TRACK`, `DAMAGE_WITHDRAW_RECOMMENDED`, … (the exact order each fires is in the [engagement-pipeline gate chain](engagement-pipeline.md#the-gate-chain-exact-order)) |
| `Logistics` | string codes | — | Strike/ferry reachability & basing (`STRIKE_UNREACHABLE`, `FERRY_UNREACHABLE_FUEL`, …) |
| `Sensor` | string codes | — | Detection/datalink health (`SENSOR_OFFLINE`, `DATALINK_STALE`, `TRACK_STALE`) |
| `Cyber` | string codes | — | Cyber/EW effects (`CYBER_LINK_DOWN`, `CYBER_ORDER_DELAY`, `CYBER_SPOOF_TRACK`) |

An **enum-mapped** family pairs each `logCode` with a C# enum `member`; a **string-code**
family lists bare codes (no enum). A single `logCode` may appear in more than one family
(e.g. `ROE_WEAPONS_TIGHT` is both a `Doctrine` and an `Engage` code, `NO_FIRE_CONTROL_TRACK`
is in both), because the same real-world reason can be reached at either layer.

## Two generated / loaded surfaces

The manifest is consumed two ways — **both must stay in sync with it**:

1. **Compile-time constants** —
   [`AbortReasonCatalog.Generated.cs`](../../src/ProjectAegis.Sim/Glossary/AbortReasonCatalog.Generated.cs)
   exposes `AbortReasonCatalog.<Family>.<CODE>` string constants (e.g.
   `AbortReasonCatalog.Engage.OUT_OF_ENVELOPE`). This is `// <auto-generated>` — **never edit
   it by hand.** Regenerate with the codegen tool (below). Consumers reference these constants
   instead of typing raw strings, so a renamed code is a compile error rather than a silent
   mismatch (tests, `MessageLogProjection`, `EngagePreviewProjection`,
   `ScenarioCyberStatusCommand`).

2. **Runtime enum → code mapping** —
   [`AbortReasonManifest`](../../src/ProjectAegis.Sim/Glossary/AbortReasonManifest.cs) parses
   the JSON at runtime (`LoadFromEmbeddedOrFile`, walking up from the assembly/CWD to find
   `data/glossary/…`). [`EngagementAbortReasonCodes.ToLogCode(reason)`](../../src/ProjectAegis.Sim/Engage/EngagementAbortReasonCodes.cs)
   uses it to turn an `EngagementAbortReason` into its `Engage`-family log code. Two special
   values are not manifest entries: `Launched` (the non-abort success row) and `NoResult`
   (queued engagement with no resolver result).

## The two-layer enum split

A fire request can be refused at **two different layers**, and each has its own enum. Keeping
them distinct is deliberate — the layers carry different information — so the engagement
resolver **maps** a policy-layer refusal onto the correct engage-layer reason rather than
collapsing everything to a generic denial.

```
Policy / ROE layer            Engagement pipeline           Log code (Engage family)
FireAbortReason               EngagementAbortReason
─────────────────             ─────────────────────         ────────────────────────
RoeHoldFire         ─┐        RoeHoldFire                   ROE_HOLD_FIRE
WeaponsTight         │        WeaponsTight                  ROE_WEAPONS_TIGHT
WraSalvo             ├─ MvpEngagementResolver.MapPolicyDenial
WraRange             │        OutOfEnvelope                 OUT_OF_ENVELOPE   ← not HOLD_FIRE
EmconOff             │        EmconOff                      EMCON_OFF
NoFireControlTrack  ─┘        NoFireControlTrack            NO_FIRE_CONTROL_TRACK
AirAspectBlock …    ── MvpEngagementResolver.MapDomainDenial → *_ASPECT_BLOCK / DOMAIN_NO_SOLUTION
```

The mappings live in
[`MvpEngagementResolver.MapPolicyDenial` / `MapDomainDenial`](../../src/ProjectAegis.Sim/Engage/MvpEngagementResolver.cs).
The important non-obvious case: **`WraRange` maps to `OUT_OF_ENVELOPE`, not `ROE_HOLD_FIRE`**
— a range-based Weapons Release Authority denial is a geometry problem, and mislabelling it as
a hold-fire order confuses AAR. This is locked by
[`DoctrineAbortReasonMappingTests`](../../src/ProjectAegis.Sim.Tests/Glossary/DoctrineAbortReasonMappingTests.cs).

### ROE denials that never reach the resolver

Under `ROE = WeaponsTight` an engage intent is rejected at the agent/policy layer *before* it
reaches `MvpEngagementResolver`, so it would produce a `POLICY_DENIAL` row but **no**
`Engagement|` abort row. To keep the order-log contract complete,
[`SimulationSession.SurfaceRoePolicyDeniedEngagements`](../../src/ProjectAegis.Delegation/Orchestration/SimulationSession.cs)
re-emits each such denial for the tick as an `Engagement|…|ROE_WEAPONS_TIGHT` abort row
(mirroring the resolver-side mapping), while preserving the original `PolicyDenial` row. The
comms-guard path (`CommsDenied`) is intentionally left as `POLICY_DENIAL`-only.

## Where the codes surface

| Surface | Row / line | Producer |
|---------|-----------|----------|
| Order log (replay) | `Engagement|…|<CODE>` | `SimulationSession.LogEngagementResults` / `SurfaceRoePolicyDeniedEngagements` |
| Order log (replay) | `PolicyDenial` (with `FireAbortReason`) | `DecisionLog.AppendPolicyDenial` |
| C2 message log | `ENGAGE_ABORT`, `POLICY_DENIAL` | [`MessageLogProjection`](../../src/ProjectAegis.Delegation/Projection/MessageLogProjection.cs) |
| Engage preview / attack menu | pre-launch abort preview | `EngagePreviewProjection` |
| Replay goldens | pinned fingerprint fragments (e.g. `ROE_WEAPONS_TIGHT`) | asserted by `ReplayGoldenSuiteTests` / harness policy-engage tests |

Because these codes are embedded in the order-log fingerprint, **changing a code string is a
behavior change** that re-records the affected replay goldens — treat it exactly like any
other determinism-affecting edit (see the
[determinism & replay guide](determinism-and-replay.md)).

## Adding or changing an abort reason

1. **Edit the manifest** — add the `logCode` (+ `member` for enum families) to the right
   family in `data/glossary/abort_reason_manifest.json`.
2. **Add the enum member** — for a `Doctrine`/`Engage` family, add the value to
   `FireAbortReason` / `EngagementAbortReason`. The `member` name in the manifest must match
   the enum member name exactly (`AbortReasonManifest` maps by `enum.ToString()`).
3. **Regenerate the constants**:

   ```bash
   pwsh tools/glossary/Invoke-AbortReasonCodegen.ps1
   ```

   This rewrites `AbortReasonCatalog.Generated.cs` from the manifest. Do not hand-edit the
   generated file; commit both the manifest and the regenerated `.cs` together.
4. **Wire the mapping** — if the reason can originate at the policy layer, add its case to
   `MvpEngagementResolver.MapPolicyDenial` / `MapDomainDenial` so it maps to the intended
   engage-layer reason (do **not** let it fall into the `_ =>` default and collapse to
   `RoeHoldFire` / `DomainNoSolution`).
5. **Verify alignment** — the guard tests below fail if the manifest, enums, and generated
   constants drift apart.

```bash
dotnet test src/ProjectAegis.Sim.Tests/ProjectAegis.Sim.Tests.csproj \
  --filter "FullyQualifiedName~Glossary" -v minimal
```

- [`AbortReasonManifestTests`](../../src/ProjectAegis.Sim.Tests/Glossary/AbortReasonManifestTests.cs)
  — every non-`None` `EngagementAbortReason` / `FireAbortReason` has a manifest entry, and
  `ToLogCode` matches the manifest for the `Engage` family; the `Logistics`/`Sensor`/`Cyber`
  string families align with the generated constants.
- [`DoctrineAbortReasonMappingTests`](../../src/ProjectAegis.Sim.Tests/Glossary/DoctrineAbortReasonMappingTests.cs)
  — policy denials round-trip to the correct, unambiguous engage code (the `WraRange` case).

If you changed a code string that appears in a pinned golden, re-record only the affected
isolated goldens and say so in the commit message (see the golden workflow in the determinism
guide).
