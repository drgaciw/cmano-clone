## 1. Problem statement

Project Aegis has a production-grade headless simulation core (deterministic tick loop, validated command API, 75+ read-only projections, DecisionLog audit) but no player-facing UI beyond a smoke-test scene. Without a UI, the game is unplayable by its target player — a human commander directing AI subordinate agents — and the project's differentiator (fluid human/AI control authority over the same command channel) remains invisible and untested. Every week without a playable surface defers validation of the core product bet.

## 2. Product identity (context)

**"Command like a commander."** Per ADR-010 the UI is a command-emitting client over the same validated command API AI agents use; the human is architecturally just another agent. The UI's distinctive job is to make that symmetry legible and safe. The core primitive is **fluid control authority**: any unit, mission, side, or authoring task can be held by the human or an AI agent and swapped live with freeze-and-hold semantics.

## 3. Goals

1. **Playable end-to-end:** a player can load an approved scenario, command the blue force through the glass cockpit, and reach a scenario conclusion without touching the CLI.
2. **Authority legibility:** in playtests, the player can state who controls any given unit (human or AI profile) within 5 seconds of being asked, using only on-screen information.
3. **Command-path symmetry proven:** an equivalent order issued via typed order line, direct manipulation, or AI agent produces the identical canonical command and outcome (UI-05), demonstrated by integration test.
4. **Determinism preserved:** replaying any scenario+seed reproduces the existing golden hash regardless of UI interaction (UI-11) — zero UI-induced determinism regressions in CI.
5. **Auditability:** for any AI-issued order, the player can reach its DecisionLog intent trace in one click (UI-04).

## 4. Non-goals

- **Watch-officer, dual-seat, and conn UI spines** — set aside at brainstorm; the attention panel preserves the oversight idea inside the cockpit. Revisit post-MVP.
- **Full direct-manipulation scenario authoring** — the editor is review-first; hand-placement UX is a separate initiative.
- **Cesium globe / APP-6 symbology as MVP scope** — phased as ADR-007 B/C. (Note: `App6GlyphAtlas` and `CesiumBillboardProjection` groundwork has partially landed, de-risking the P2 line; MVP still ships the Phase A map.)
- **Multiplayer or spectator/broadcast modes** — out of scope; architecture must not preclude them.
- **Gamepad support** — **recorded exception to the ui-experience-lead gate** ("keyboard/mouse and gamepad navigation are explicit"): descoped for MVP with the mitigation that all navigation is built on UI Toolkit's navigation-event system (see NFR-2), so gamepad becomes additive, not a rework.
- **Text scaling beyond 100%** — explicitly deferred with rationale: the MVP cockpit is a fixed-density layout validated at 1600×900 @ 100%; OS-scale (125%+) support is revisited with the post-MVP scaling design rather than half-supported now.

## 5. User stories

**Persona: Commander (human player, default posture: direct control)**

- As a commander, I want every cockpit panel (units, contacts, map, weapons, missions, log, attention) always visible so that I maintain situational awareness without window management. (UI-01)
- As a commander, I want each unit, mission, and the blue-side default to show its effective owner and AI profile so that I always know who has the conn. (UI-02)
- As a commander, I want to take control of any AI-held unit and have its last accepted orders preserved so that handoff never destroys work in progress. (UI-03)
- As a commander, I want to issue an order by typing it or by direct manipulation and get the same validated result either way so that I can use whichever is faster. (UI-05)
- As a commander, I want a rejected order to show its reason and keep my draft and selection so that I can correct it without reconstructing intent. (UI-06)
- As a commander, I want the sim to pause on new contacts and own-side losses with a visible reason so that critical moments never pass unseen. (UI-08)
- As a commander, I want an AI weapons-release request to drop compression to 1x with a DLZ-expiry countdown so that I can approve or deny before the launch window closes.
- As a commander, I want to click any AI order and see why the AI did it so that I can calibrate my trust in delegation. (UI-04)

**Persona: Delegating commander (same player, AI-operator mode)**

- As a delegating commander, I want to hand the whole blue side to an AI operator and take it back at will so that I can run at high compression and intervene by exception.
- As a delegating commander, I want side-level handoff to preserve my explicit unit/mission assignments (most-specific wins) so that delegation never silently overwrites my decisions.
- As a delegating commander, I want attention cards with new/acknowledged/resolved states and deterministic ordering so that triage is reliable under load. (UI-07)

**Persona: Scenario reviewer (human, reviewing hand- or agent-authored scenarios)**

- As a scenario reviewer, I want to load a scenario, see its ADR-008 validation report and map preview, tweak it, and approve it so that only vetted scenarios reach play. (UI-09)
- As a scenario reviewer, I want approval to be a reversible command carrying full ADR-015 provenance so that trust decisions are auditable and undoable.
- As a scenario reviewer, I want approval blocked while validation reports blocking errors so that broken scenarios cannot slip into rotation. (UI-09)

**Edge/negative stories**

- As a commander, when my handoff request is pending, I never see a mixed-owner state; it resolves to exactly one accepted or rejected result. (UI-03, §3.9)
- As a commander, acknowledging, filtering, or dismissing attention cards never alters simulation state or discards the underlying record. (UI-07)

### Cross-cutting NFRs (P0)

- **NFR-1 Localization:** all player-facing strings externalized from day one — no hardcoded strings (ui-experience-lead gate). Sim/delegation-emitted text (rejection reasons, validation errors, DecisionLog summaries) is surfaced as **localizable keys + parameters**, not literal strings (see OQ-3). Translated locales are P1.
- **NFR-2 Input architecture:** all focus/navigation built on UI Toolkit's navigation-event system, not raw key handlers — the enabling condition for the gamepad descope exception.
- **NFR-3 UI performance:** log/roster/attention use virtualized ListView; all bindings are change-driven off projection updates — **no per-frame queries of gameplay state**; attention sort computed in the controller (`C2PresentationController` pattern).

### 6.2 Nice-to-have (P1)

- **Contact-pause grouping UI:** per-scenario option to group new-contact pauses by raid/formation with a cooldown window (the P0-7 grouping keys make this a filter).
- **Attention filters by category/priority** beyond the P0 restorable dismiss.
- **AI guidance-request quick responses:** canned replies (proceed per doctrine / hold / RTB) on guidance cards.
- **Order-line autocomplete** from the command grammar.
- **Translated locales** (externalization itself is P0, NFR-1).

### 6.3 Future considerations (P2)

- Watch-officer, dual-seat, and conn layout modes over the same projections/commands (architectural insurance: keep layout separate from binding).
- Cesium globe + APP-6 symbology (ADR-007 B/C) — groundwork partially landed (`App6GlyphAtlas`, `CesiumBillboardProjection`).
- Full direct-manipulation scenario authoring on the review-first foundation.
- Spectator/broadcast + AAR timeline scrubbing over the replay/order log.
- Red-force expertise adjustment mid-run (MVP: pre-run scenario setting only).
- Text scaling / OS display-scale support (see Non-goals).

## 7. Success metrics

**Leading (playtest, first 4 weeks of playable builds)**

- Time-to-first-accepted-order in a fresh scenario: ≤ 2 minutes without documentation (stretch: ≤ 1).
- Authority legibility probe: correct "who controls this unit?" answers within 5 seconds in ≥ 90% of probes.
- Order-path symmetry usage: both typed and direct-manipulation paths used in ≥ 80% of playtest sessions.
- Attention hygiene: ≥ 70% of attention cards resolved via command or terminal condition (vs. dismissed) in a standard scenario.
- Pause tolerance: player disables or requests changes to auto-pause in < 30% of sessions; pauses-per-minute from the P0-7 dense-scenario walkthrough is the early-warning measure. Breach → escalate P1 grouping UI to fast-follow.

**Lagging (per release)**

- Zero UI-induced golden-hash regressions in CI (P0-11 harness) — hard gate, not a target.
- Scenario completion rate in playtests ≥ 60%.
- 100% of AI orders traceable to a DecisionLog intent trace (UI-04 coverage).
- Editor: ≥ 90% of agent-generated (qa-gauntlet) scenarios pass through review → approve/reject without CLI fallback.

## 9. Timeline considerations (recommended phasing)

No hard external deadlines; constraints are solo-dev/agent-team throughput and the ADR queue. Each phase independently shippable and testable headless-first; deliver via the repo's Graphite stacked-PR flow.

1. **Phase 1 — Command spine:** P0-4 (log extension), P0-5 (order line over existing bridge, new grammar for 5 OrderKinds), NFR-1 string externalization, start of P0-11 harness. Cheapest end-to-end slice; proves symmetry; unblocks all panels. Needs OQ-3 early.
2. **Phase 2 — Cockpit + authority:** P0-1, P0-2, P0-3, P0-12, P0-10, NFR-2/3. **Gated on OQ-1 (authority model) and OQ-2 (wireframe sign-off).**
3. **Phase 3 — Time + attention + engagement:** P0-6, P0-7, P0-8. All three carry [new] sim/delegation work (attention event source, grouping keys, time-to-DLZ-exit, approve/deny commands) — design those headless contracts first; wireframe the attention card before building.
4. **Phase 4 — Editor review flow:** P0-9. Gated on OQ-5; can run in parallel with Phase 3 (touches validation/approval paths, not the play-time command path).

**Scope rule:** any addition to P0 requires removing something from P0 or explicitly extending the phase — the Non-goals list is the parking lot.
